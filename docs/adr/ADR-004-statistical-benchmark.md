# ADR-004: ベンチマーク採用判定に統計的閾値を使う

- **日付**: 2026-06-19
- **状態**: Accepted

## 背景

GPU ベンチマークは測定ノイズが大きく、GPU クロック・温度・他プロセス・JIT コンパイル等の影響で
同じカーネルを 2 回計測しても 1〜2% の差が普通に出る。単純な 1 回比較では誤採用・誤棄却が頻発する。

## 決定

採用判定を以下の条件で行う：

```python
def is_improvement(candidate, baseline, min_speedup=1.03):
    return candidate.p80_us < baseline.p20_us / min_speedup
```

- ウォームアップ 25 回 + 測定 200 回
- 採用閾値は **3% 以上の改善**（1% では測定ノイズと区別できない）
- `p80(candidate) < p20(baseline) / 1.03` で「候補の遅い側が baseline の速い側より速い」を要求

## 理由

- 1% 閾値はノイズレベルと被る。3% を下回る改善は実用上意味が薄い
- p80/p20 の比較は「ほぼ確実に速い」という保守的判定。中央値同士の比較より信頼性が高い
- 200 サンプルは GPU ベンチマークの標準的な反復数として十分な統計量を確保できる

## トレードオフ

- 3% 未満の真の改善を見逃す可能性がある（保守的すぎる）
  → 最初はこれで良い。将来、`--strict` / `--relaxed` フラグで閾値を調整可能にする
- 200 回 × 候補数 分のベンチマーク時間がかかる（50 候補で ~10 分）
  → 2 段階評価（粗い測定で絞り込み → 上位候補のみ精密測定）で緩和予定（Phase 4 以降）
