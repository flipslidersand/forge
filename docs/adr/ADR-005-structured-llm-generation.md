# ADR-005: LLM には自由コード生成ではなく構造化命令を出させる

- **日付**: 2026-06-19
- **状態**: Accepted

## 背景

MVP 3 で LLM を候補生成に導入する際、「Triton コードを自由に書かせる」か「構造化パラメータを出させる」かの選択が必要。
自由コード生成は表現力が高いが、検証コスト・再現性・デバッグのしやすさで問題が生じやすい。

## 決定

LLM には以下の JSON 形式で候補を出力させる：

```json
{
  "base_variant": "single_row",
  "params": {
    "block_size": 4096,
    "num_warps": 8,
    "acc_dtype": "fp32",
    "rows_per_program": 2
  },
  "hypothesis": "hidden_size=4096 では block_size=4096 が L2 キャッシュに収まる"
}
```

実際の Triton コードはテンプレートエンジンが生成する。LLM はパラメータと仮説のみを担当する。

## 理由

- 検証しやすい：パラメータの範囲バリデーションのみで済む
- 再現可能：同じ JSON → 同じコード → 同じ実行結果
- 差分が明確：`hypothesis` フィールドで LLM の推論を記録できる
- コード破壊がない：LLM が構文エラーを出しても探索が止まらない
- 将来の自由コード生成への移行パスを残せる（`CandidateGenerator` Protocol）

## トレードオフ

- テンプレートで表現できない最適化は LLM では試せない
  → MVP では意図的な制約。Phase 5 以降で `raw_code` フィールドを追加して段階的に拡張する
- LLM の「創造性」が制限される
  → 探索空間が広い段階では構造化の方が費用対効果が高い
